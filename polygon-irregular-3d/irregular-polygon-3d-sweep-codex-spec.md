# Irregular Polygon 3D Sweep Applet — Codex Implementation Specification

## Current First-Pass Implementation

The live app now treats a sequence of user-editable polygons as **structural loft sections**. Each section has a position along the selected sweep path, independent scale, rotation, and local X/Y offset; all sections contribute real mesh rings. The generator resamples and phase-aligns their boundaries before interpolating between adjacent sections. This supports enclosed, mineral-like forms without reducing the object to a single central belt.

Guide-only polygons are intentionally deferred. In this pass every polygon station is structural and directly affects the generated mesh.

The app also includes an editable X/Z path mode: users can place, drag, insert, and delete path points, then choose a smooth Catmull-Rom spline or a straight polyline. Section positions continue to resolve by normalized distance along that authored path.

Tight bends now expose an explicit collision policy: preserve the authored intersecting loft, apply local curvature-based scale clearance, or rasterize the actual loft into a signed interior/exterior field and extract a fused SDF patch only around detected collision rings. Original source triangles outside those collision zones remain in the rendered/exported mesh unchanged.

## 1. Project Summary

Build a self-contained browser applet that allows a user to:

1. Draw and edit an irregular closed polygon in 2D.
2. Use that polygon as the cross-section of a larger 3D object.
3. Select a creation mode.
4. Adjust geometry through sliders and toggles.
5. Preview the result interactively in 3D.
6. Export the generated mesh.

The primary construction method is a **cross-section sweep**: the user-drawn polygon is sampled repeatedly along a path, transformed at each sample, and connected into a continuous 3D mesh.

The app should also expose simpler geometric modes such as extrusion, taper, twist, helix, arc, and loft as presets or constrained versions of the same sweep engine.

The initial deliverable should run locally by opening `index.html` or through a lightweight static server. Avoid any backend requirement.

## 2. Core Product Goal

The app should make it possible to draw an arbitrary polygon and rapidly explore how it behaves as the cross-section of a three-dimensional form.

The user should be able to create forms such as:

- straight prisms;
- tapered extrusions;
- twisted prisms;
- curved horns;
- helical tubes;
- shells;
- ring-like forms;
- ribbed or pulsing structures;
- forms generated by changing scale, rotation, and offset along a path.

The system should preserve the relationship between the original polygon and the resulting mesh. Cross-section rings should remain available as an optional visual overlay.

## 5. Primary Geometry Model

Let the user-drawn polygon be:

\[
P = \{p_0,p_1,\ldots,p_{n-1}\}
\]

where:

\[
p_i=(x_i,y_i)
\]

Let the sweep path be:

\[
\gamma(t), \qquad t\in[0,1]
\]

At each sampled path position \(t_k\), place a transformed copy of the polygon:

\[
V_{i,k}
=
\gamma(t_k)
+
s_x(t_k)x_iN(t_k)
+
s_y(t_k)y_iB(t_k)
\]

where:

- \(N(t)\) is the local normal-axis direction;
- \(B(t)\) is the local binormal-axis direction;
- \(s_x(t)\) is horizontal cross-section scale;
- \(s_y(t)\) is vertical cross-section scale.

Apply user twist \(\theta(t)\) by rotating the local \(N/B\) basis around the tangent direction.

Optional offsets should modify the ring center:

\[
C_k
=
\gamma(t_k)
+
o_x(t_k)N(t_k)
+
o_y(t_k)B(t_k)
\]

Then:

\[
V_{i,k}
=
C_k
+
s_x(t_k)x_iN_\theta(t_k)
+
s_y(t_k)y_iB_\theta(t_k)
\]

Adjacent rings are connected with quads:

\[
(V_{i,k},V_{i+1,k},V_{i+1,k+1},V_{i,k+1})
\]

Convert each quad into two triangles.

Use consistent winding order so surface normals point outward.

## 11. Path Orientation and Frames

### 11.1 Requirement

Do not orient cross-sections using only a naive Frenet frame.

Use a **parallel-transport frame** as the default to reduce sudden flips and unintended twisting.

### 11.2 Parallel-Transport Outline

For sampled path points:

```text
C[0], C[1], ..., C[m]
```

Compute tangent vectors:

```ts
T[k] = normalize(C[k + 1] - C[k])
```

Choose an initial normal vector not parallel to `T[0]`.

For each next tangent:

1. compute the rotation axis:

```ts
axis = cross(T[k - 1], T[k])
```

2. compute the angle:

```ts
angle = atan2(length(axis), dot(T[k - 1], T[k]))
```

3. rotate the previous normal around `axis`;
4. recompute the binormal:

```ts
B[k] = normalize(cross(T[k], N[k]))
```

5. recompute the corrected normal:

```ts
N[k] = normalize(cross(B[k], T[k]))
```

Handle nearly parallel tangents without introducing unstable rotations.

### 11.3 Closed Paths

For closed paths, accumulated orientation error may cause the final frame to differ from the first.

Implement a closure correction:

1. calculate residual rotation between the final and initial frames;
2. distribute the inverse residual rotation gradually across all rings.

Resampling is especially important for:

## 13. Mesh Construction

### 13.1 Side Faces

For ring count \(m\) and section vertex count \(n\):

```text
vertex index = ringIndex * n + sectionIndex
```

For each adjacent ring pair and section edge:

```ts
a = index(k, i)
b = index(k, (i + 1) % n)
c = index(k + 1, (i + 1) % n)
d = index(k + 1, i)
```

Triangles:

```text
(a, b, d)
(b, c, d)
```

Reverse winding if polygon orientation or frame convention requires it.

### 13.2 Caps

For open paths, optionally cap the first and last rings.

Use a robust triangulation library such as:

```text
earcut
```

Cap triangulation should support concave simple polygons.

The end cap winding must be opposite the start cap winding.

### 13.3 Normals

After creating the indexed buffer geometry:

```ts
geometry.computeVertexNormals()
```

Support:

- smooth shading;
- flat shading.

For flat shading:

```ts
geometry.toNonIndexed()
geometry.computeVertexNormals()
```

### 13.4 UV Coordinates

UVs are optional for the first version.

Recommended later implementation:

- \(u\): cumulative perimeter position around the cross-section;
- \(v\): normalized distance along the path.

## 30. Hrifa-Oriented Extension Model

Keep the engine general enough that later systems can drive sweep parameters procedurally.

Potential mapping:

```text
Material current      → lower noise, stronger regularity, reduced taper variance
Temporal current      → twist rate, oscillation, cyclic modulation
Transitional current  → discontinuities, stepped changes, fractures, abrupt scale shifts
Symbolic current      → repeated motifs and quantized rotation
Ontological current   → enclosure, shell thickness, cavity formation
Cognitive current     → path redirection or focus-seeking curvature
Interpersonal current → branching, pairing, convergence, or reciprocal paths
```

Suggested future procedural interface:

